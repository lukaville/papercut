# Manual Builder

A web app for authoring **step-by-step assembly manuals** for papercut paper
models. Each step shows the model in an isometric (or custom) view, with the
already-built parts in place and the newly added parts pulled out along an
explode direction, connected back to their attachment points with dashed leader
lines — an editable exploded view.

It runs entirely in the browser. There is **no backend**: project directories
are read and written directly through the Chrome/Edge
[File System Access API](https://developer.mozilla.org/en-US/docs/Web/API/File_System_API).

---

## Quick start

```bash
# 1. Generate the 3D model data for a project (uses papercut-processor / CadQuery)
./process projects/cut_sample
#    …or, non-destructively, just the manual model from cached geometry:
PYTHONPATH=papercut-processor papercut-processor/.venv/bin/python \
  papercut-processor/export_manual.py projects/cut_sample

# 2. Run the app
cd manual-builder
npm install
npm run dev          # http://localhost:5174
```

Open the app, click **Open project directory**, and pick either a single
project folder (e.g. `projects/cut_sample`) or the `projects/` root (you'll then
choose a sub-project). Grant read/write access when prompted. The opened project
and your UI state are restored automatically on refresh.

> **Browser support:** the File System Access API is available in Chromium
> browsers (Chrome, Edge, Brave, Arc). Firefox/Safari are not supported.

## On-disk format (`[project]/manual/`)

```
[project]/manual/
├── manual.json            # authored content — commit this
└── model/                 # generated by papercut-processor — regenerable
    ├── model.json         # parts catalog + instance placements
    └── meshes/
        └── <partKey>.json # geometry + topological vertices, per unique part
```

`model/` is **generated** (re-run the processor after CAD changes).
`manual.json` is **authored** by this app and is the thing worth version
controlling.

### `model/model.json` (read-only, produced by Python)

```jsonc
{
  "schemaVersion": 1,
  "thickness_mm": 0.5,
  "bounds": { "min": [x,y,z], "max": [x,y,z] },
  "parts": [
    {
      "key": "outer_front",        // STABLE 3D part name — the anchor
      "color": "#559fd7",
      "mesh": "meshes/outer_front.json",
      "vertexCount": 104,
      "bbox": { "min": [...], "max": [...] }
    }
  ],
  "instances": [
    {
      "id": "outer_front#0",       // STABLE id = partKey + ordinal
      "partKey": "outer_front",
      "matrix": [ /* 16 floats, column-major, local→world */ ],
      "sheet": "A12"               // display-only; may change between runs
    }
  ]
}
```

Each `meshes/<partKey>.json` holds `positions`/`indices` (the render mesh) plus
`vertices` — the part's **topological vertices in canonical order**. A vertex is
referenced by its index into that array.

### `manual.json` (authored here)

```jsonc
{
  "schemaVersion": 1,
  "title": "Assembly Manual",
  "defaults": {
    "upAxis": "z",
    "view": { "azimuthDeg": 45, "elevationDeg": 35.264, "zoom": 1 }
  },
  "steps": [
    {
      "id": "uuid-a",
      "description": "Front wall",
      "view": null,                // null → inherit the default view
      "dependsOn": [],             // independent — starts from scratch
      "added": ["outer_front#0"],  // new parts introduced this step
      "connections": []
    },
    {
      "id": "uuid-b",
      "description": "Add the inner wall",
      "view": null,
      "dependsOn": ["uuid-a"],     // builds on step A (its parts are context)
      "added": ["inner_front#0"],
      "parts": {                   // explode is per-part and on-demand
        "inner_front#0": {
          "explode": { "distance_mm": 60, "direction": [0, 0, 1] }
        }
      },
      "connections": [
        { "id": "uuid",
          "from": { "instance": "inner_front#0", "vertex": 2 },
          "to":   { "instance": "outer_front#0", "vertex": 5 } }
      ]
    }
  ]
}
```

**Step context is dependency-based.** A step's *existing* (already-built) parts
are derived as the union of the visible parts of every step it `dependsOn`,
resolved transitively (with a cycle guard). This means:

- A new step **depends on the previous one by default** (so it carries prior
  work forward as ghosted context), but you can make an **independent** step
  (`dependsOn: []`) to focus on a subsection from scratch, or depend on
  **several** steps to merge subassemblies.
- Dependencies are by **step id**, so reordering steps never silently changes
  what a step shows.

**Explode is per-part and on-demand**: a new part is only offset if it has its
own `parts[id].explode`; there is no step- or document-level default explode.

## Robustness to CAD changes

The manual is meant to survive edits to the source model. Three anchoring
choices make that possible (all best-effort, by design):

1. **Parts are keyed by 3D part name** (`partKey`), never by sheet labels like
   `A12` — sheet labels are reassigned on every nesting run.
2. **Instances use a stable id** `"<partKey>#<ordinal>"`, where the ordinal is
   derived from a *coarsely rounded* world position, so unrelated edits don't
   renumber existing instances.
3. **Vertices are referenced by index** into the part's topological vertex list
   (OCC sub-shape order). This survives dimensional changes as long as the
   part's topology is unchanged.

References that can no longer be resolved (a removed part, a vanished vertex) are
skipped gracefully rather than breaking the view.

## Architecture

```
src/
├── fs/            File System Access layer
│   ├── pick.ts            directory picker, project detection, permissions
│   ├── handleStore.ts     IndexedDB persistence of the directory handle
│   └── projectSource.ts   typed read/write of model + manual
├── store/         Zustand store (lifecycle, mutations, debounced autosave)
├── types/         shared schema (model.ts, manual.ts) + helpers
├── lib/           pure math (vec, camera, geometry, step resolution)
├── viewport/      react-three-fiber: freeform Viewport + persistent StepView,
│                  shared MeshProvider, camera rig, parts, connectors
├── panels/        Steps, Step inspector, Part inspector, Parts, Options (each a dockable panel)
├── components/    toolbar, project gate, form fields
└── layout/        FlexLayout (caplin) docking shell
```

- **React + TypeScript + Vite**, strict mode.
- **[caplin/FlexLayout](https://github.com/caplin/FlexLayout)** for the IDE-style
  dockable panels (your layout is persisted to `localStorage`).
- **react-three-fiber + drei** for the orthographic 3D view.
- **Zustand** for state; the manual is mutated immutably and **autosaved**
  (debounced, atomic write) to `manual.json`.
- Persistence across refreshes: the directory handle lives in **IndexedDB**;
  the opened project name, selected step and UI flags live in `localStorage`.

### Authoring workflow

1. **Open** a project (must already have `manual/model/`).
2. **Add a step** (depends on the current step by default) or **+ Independent**
   (no dependencies, from scratch). In the **Step inspector → Depends on** you
   can change which step(s) it builds upon — including several, to merge
   subassemblies. Inherited context parts show ghosted, in place.
3. In the **Parts** panel (grouped *new → existing → other*) hit **+ new** to add
   instances to the step; existing context parts are listed but come from
   dependencies.
4. Two viewports: the center **freeform viewport** (orbit freely) and the small,
   non-interactive **Step view** (top-right) that always shows the step's *saved*
   view. Orbit, then **⤓ Save as step view**; **⤒ Match step view** snaps the
   freeform camera back. You can also type exact azimuth/elevation/zoom in the
   Step inspector.
5. Click a new part (in either viewport or the Parts panel) and, in the **Part
   inspector**, toggle **explode** to give it an on-demand offset
   (distance + direction). Parts are not exploded by default.
6. Toggle **connect mode**, then click a vertex on a new part and a vertex on a
   built part to draw a dashed leader line between them.

### Viewport controls (macOS trackpad)

The freeform viewport is tuned for the trackpad ([TrackpadControls.tsx](src/viewport/TrackpadControls.tsx)):

| Gesture                     | Action          |
| --------------------------- | --------------- |
| Drag (one- or two-finger click-drag) | Rotate |
| Two-finger swipe            | Pan             |
| Pinch                       | Zoom (to cursor) |

The browser delivers both two-finger pan and pinch as `wheel` events (pinch sets
`ctrlKey`); OrbitControls' built-in wheel zoom is disabled and these are handled
explicitly so panning no longer zooms. Rotation uses damping for a smooth feel.

## Producing model data (Python side)

The 3D data is produced by `papercut-processor`:

- `manual_exporter.py` — `export_manual_model(...)` writes `manual/model/`. It is
  called automatically at the end of `./process <project>`.
- `export_manual.py` — a standalone, **non-destructive** CLI that rebuilds only
  `manual/model/` from cached geometry (does not touch `parts/`, `sheets/`,
  `overlays/`). Handy while iterating on the app.

Both share `naming.py` so part keys are identical to the rest of the pipeline.

## Out of scope (intentionally) / future work

The brief deliberately leaves "manual-like" output for later. Natural next
features that this data model already supports:

- **Page layout & export** — paginate steps, add captions/part lists, export to
  PDF/SVG/PNG, with per-manual color and styling themes.
- Staged sub-explosions / animated explode transitions.
- Auto-suggested connections (nearest coincident vertices between new and built
  parts).
- Multiple manuals per project (the format is one document today; the store is
  abstracted enough to grow into a documents list).
